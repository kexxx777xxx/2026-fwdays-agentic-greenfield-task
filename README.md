# askdocs

**Чат з документацією** — локальний RAG-інструмент. Ставиш питання, отримуєш
відповідь, що спирається **тільки** на твій локальний корпус `.md`-файлів, із
посиланням на файл-джерело. Якщо відповіді в документації немає — askdocs чесно
каже про це, а не вигадує.

> Це домашній проєкт курсу fwdays Academy. Умови завдання — у
> [docs/HOMEWORK.md](docs/HOMEWORK.md). Продуктова мапа — у [docs/prd.md](docs/prd.md).

## Навіщо це

Моделі впевнено вигадують. Для питань по внутрішній документації це небезпечно:
краще почути «цього в документації немає», ніж правдоподібну неправду. askdocs
відповідає лише з наданого корпусу і завжди показує джерело — відповідь без
джерела вважається багом.

Тестовий корпус — синтетичний домен **ГущоЛіт** (вигаданий проєкт літаючих авто
на кавовій гущі). Домен навмисно вигаданий: правильну відповідь можна дати лише
з корпусу, а не із «загальних знань» моделі. Це робить перевірку чесності
однозначною.

## Архітектура

Три обов'язкові інтерфейси, над якими не знає про конкретні реалізації решта коду:

- **`DocSource`** — «дай документи». v1: локальні `.md`.
- **`Retriever`** — «знайди релевантне під питання». v1: векторний пошук.
- **`LLMProvider`** — «згенеруй відповідь з контексту». v1: одна модель.

Конвеєр: `ingest` (корпус → чанки → vector store) → `retrieve` (питання → чанки)
→ `answer` (чанки + питання → відповідь із цитуванням або чесне «не знаю»).
`sync` тримає індекс актуальним, `eval` міряє якість, `cli` — інтерфейс.

Vector store (**qdrant**) — окремий сервіс. Embedding — локальний
`sentence-transformers` (multilingual). LLM — будь-який OpenAI-сумісний ендпоінт
(за замовчуванням LM Studio на хості).

## Передумови

Щоб **запускати й тестувати** askdocs, потрібен лише **Docker** (з
docker-compose) — увесь код і залежності живуть у контейнері, у системний
Python хоста нічого не ставиться. Для відповідей (`answer`, `cli`) потрібен
локальний LLM — за замовчуванням [LM Studio](https://lmstudio.ai) на хості з
OpenAI-сумісним API на `http://localhost:1234/v1` (модель — через `LLM_MODEL`).
`ingest`, `retrieve`, `sync` і тести LLM не потребують.

Щоб **вести розробку по специфікаціях**, встанови OpenSpec CLI (проєкт ведеться
через нього — див. [AGENTS.md](AGENTS.md)):

```bash
brew install openspec         # або: npm install -g openspec  (див. https://openspec.dev)

# Згенерувати slash-команди/скіли воркфлоу під свій AI-асистент.
# Замість claude постав свій тул: cursor, codex, gemini, windsurf, cline, …
openspec init --tools claude
```

`openspec init` створює обгортки в `.claude/` (чи в теці твого тула) — вони
згенеровані й не комітяться, тож кожен генерує їх під себе. Самі специфікації і
зміни (`openspec/specs`, `openspec/changes`) — у репозиторії. OpenSpec потрібен
лише для роботи зі специфікаціями; щоб просто запустити продукт, досить Docker.

## Запуск

Усе живе в Docker — у системний Python хоста нічого не ставиться.

```bash
# Підняти всю систему: qdrant + сервіс, що індексує ./corpus і стежить за змінами
docker compose up
```

Кидай свої `.md` у `./corpus` — вони автоматично переіндексуються (додавання,
редагування, видалення).

### Поставити питання

```bash
docker compose run --rm app python -m askdocs.cli "Який сорт гущі заборонений як паливо?"
```

Відповідь виводиться разом зі списком файлів-джерел. Без аргументу —
інтерактивний режим (питання-відповідь у циклі).

Для відповідей потрібен LLM: за замовчуванням askdocs звертається до LM Studio
на хості (`http://localhost:1234/v1`). Модель налаштовується змінною `LLM_MODEL`.
`ingest`, `retrieve` і `sync` LLM не потребують.

### Ручний ingest (без watcher)

```bash
docker compose run --rm app python -m askdocs.ingest <тека-корпусу>
```

## Тести і eval

```bash
# Уся тестова батарея (у контейнері)
docker compose run --rm app pytest

# Звіт eval: retrieval hit-rate + anti-hallucination refusal-rate по ГущоЛіт
docker compose run --rm app python -m askdocs.eval
```

Тести, що потребують LLM (генерація відповіді, anti-hallucination), автоматично
пропускаються, якщо ендпоінт недоступний — решта батареї лишається зеленою.
