# Проєкт: askdocs

RAG-інструмент "чат з документацією". Відповідає на питання по локальному
корпусу .md, спираючись ТІЛЬКИ на нього, з посиланням на джерело.

## Домен (тестовий корпус)

ГущоЛіт — вигаданий проєкт літаючих авто на кавовій гущі. Домен навмисно
синтетичний: правильна відповідь може прийти лише з корпусу, не зі знань
моделі. Якщо модель відповідає з "загальних знань" — це помилка.

## Стек

Python. Один embedding-провайдер, один vector store, одна LLM — усі три
за інтерфейсами (див. нижче). Без reranker'ів, графів, агентних надбудов.

## Оточення (Docker)

Увесь застосунок і його залежності живуть у Docker, НЕ в системному python
хоста. Оркестрація через docker-compose:

- сервіс `app` — python + код askdocs;
- сервіс vector store — окремий образ (конкретний обирається в design.md).

Будь-який запуск (`ingest`, `cli`, `pytest`) відбувається всередині
контейнера, напр. `docker compose run --rm app pytest`.
Хост лишається чистим: жодних pip-встановлень у систему.

## Архітектурні межі (обов'язкові)

- `DocSource` — інтерфейс "дай документи". v1: локальні .md.
- `Retriever` — інтерфейс "знайди релевантне під питання". v1: вектори.
- `LLMProvider` — інтерфейс "згенеруй відповідь з контексту". v1: одна модель.

Код над цими інтерфейсами не знає про конкретну реалізацію.

## Правила роботи

- Реалізуй ТІЛЬКИ вимоги зі статусом `accepted`. `proposed` не чіпай.
- Критерій готовності — зелені тести, не "здається працює".
- Відповідь без джерела = баг. "Немає в корпусі" — валідна відповідь, не помилка.
- Кожен capability проводь окремим OpenSpec change.
- Vector store — окремий сервіс у docker-compose (не embedded).
- Тести ідемпотентності ingest ганяються проти чистого стану vector store
  (ephemeral volume або teardown перед прогоном), щоб перевірка була чесною.
- Нічого не встановлюй у системний python хоста — тільки в контейнер.
